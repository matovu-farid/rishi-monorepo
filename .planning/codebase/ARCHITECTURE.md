# Architecture

**Analysis Date:** 2026-04-05

## Pattern Overview

**Overall:** Multi-app monorepo with a Tauri desktop app as the primary client, a Next.js web app serving as an auth landing page, a Cloudflare Worker as the backend API, and an Expo mobile app (early-stage).

**Key Characteristics:**
- Tauri desktop app (Rust backend + React/TypeScript frontend) is the core product
- Rust backend exposes typed commands to the React layer via Tauri's `invoke` IPC bridge
- Auth is delegated: Tauri → browser (web app) → Clerk → deep-link callback → Cloudflare Worker JWT exchange
- All AI/LLM calls are proxied through the Cloudflare Worker (never direct from client)
- Local data is stored in SQLite (via Diesel) and HNSW vector files on disk

## Layers

**Tauri Rust Backend (`apps/main/src-tauri/src/`):**
- Purpose: System-level capabilities, local storage, AI embeddings, book parsing
- Location: `apps/main/src-tauri/src/`
- Contains: Tauri command handlers, SQLite/Diesel ORM, HNSW vector store, embedding engine, EPUB/PDF parsers, auth logic, LLM proxy calls
- Depends on: Cloudflare Worker (HTTP), local filesystem, Tauri plugins
- Used by: React frontend via `invoke()`

**Tauri Command Interface (generated):**
- Purpose: Type-safe TypeScript bindings auto-generated from Rust command signatures
- Location: `apps/main/src/generated/commands.ts`, `apps/main/src/generated/types.ts`
- Contains: `invoke()` wrappers for every registered Rust command
- Generated by: `cargo tauri-typegen generate` — **do not edit manually**
- Used by: React components, hooks, modules throughout `apps/main/src/`

**React Frontend (`apps/main/src/`):**
- Purpose: UI rendering, user interactions, state management, document viewing
- Location: `apps/main/src/`
- Contains: Routes, components (EPUB viewer, PDF viewer, TTS controls), Jotai atoms, modules (TTS service/queue/cache, SQL helpers), generated bindings
- Depends on: Tauri commands (via `@/generated`), Jotai store, TanStack Router, TanStack Query
- Used by: End users via the desktop app

**Cloudflare Worker (`workers/worker/src/index.ts`):**
- Purpose: Authenticated API gateway for OpenAI, Deepgram, and Clerk; handles auth token exchange and JWT issuance
- Location: `workers/worker/src/index.ts`
- Contains: Hono routes for auth exchange, TTS, realtime client secrets, LLM completions, user lookups
- Depends on: Clerk (identity verification), Upstash Redis (auth state), OpenAI, JWT signing
- Used by: Tauri Rust backend (HTTP), web app (server actions)

**Next.js Web App (`apps/web/src/`):**
- Purpose: Auth landing page for the Tauri deep-link OAuth flow; marketing/landing page
- Location: `apps/web/src/`
- Contains: Next.js App Router pages, Clerk provider, `ClerkListener` client component, server action (`lib/redis.ts`), Jotai atoms for auth state, Clerk middleware
- Depends on: Clerk (NextJS SDK), Upstash Redis (auth state write), Cloudflare Worker (indirectly)
- Used by: Tauri app (browser redirect), end users (landing page)

**Expo Mobile App (`apps/mobile/`):**
- Purpose: Mobile reading client (early stage, minimal implementation)
- Location: `apps/mobile/`
- Contains: Expo Router tab-based layout, placeholder screens
- Depends on: React Native ecosystem

## Data Flow

**Book Loading (EPUB/PDF):**
1. User drops a file → `FileComponent.tsx` calls `getBookData` / `getPdfData` via `@/generated`
2. Rust command parses the file (EPUB via `epub.rs`, PDF via `pdf.rs`), stores metadata to `store.json` via `tauri_plugin_store`
3. Parsed `BookData` returned to React; book is saved to SQLite via `sql::save_book`
4. UI navigates to `/books/$id` route, loading EPUB or PDF viewer component

**RAG / AI Q&A Flow:**
1. On book open, pages are chunked and `embed` command is called → Rust uses `embed-anything` with `all-MiniLM-L6-v2` BERT model locally
2. Embeddings saved to HNSW vector store on disk (`vectordb.rs`) and chunk text saved to SQLite
3. On user query, `getContextForQuery` fetches top-k nearest vectors, retrieves associated text from SQLite
4. `llm.rs` calls `POST /api/text/completions` on Cloudflare Worker with JWT auth, returns LLM answer

**TTS Flow:**
1. User triggers audio for a paragraph → `TTSService.requestAudio()` called in `modules/ttsService.ts`
2. Cache checked via `ttsCache.ts`; if miss, queued via `ttsQueue.ts`
3. Queue calls Cloudflare Worker `POST /api/audio/speech` with JWT, returns audio
4. Audio cached locally; `TTS_EVENTS.AUDIO_READY` emitted to listeners

**Auth Flow (Tauri → Clerk → JWT):**
1. `LoginButton.tsx` calls `getState` Rust command → generates UUID state + code_verifier, stores in `store.json`
2. Browser opens `https://rishi.fidexa.org?login=true&state=...&code_challenge=...`
3. `ClerkListener` in web app detects sign-in, calls server action `saveUser` → writes `userId` + state + codeChallenge to Upstash Redis
4. Web app redirects to `rishi://auth/callback?state=...` (deep link)
5. Tauri catches deep link via `onOpenUrl`, calls `completeAuth` Rust command
6. Rust POSTs `{ state, code_verifier }` to Cloudflare Worker `/api/auth/complete`
7. Worker verifies state from Redis, validates PKCE, issues JWT → returns `{ token, user }`
8. Tauri stores JWT and user in `store.json`; future API calls use Bearer JWT

**State Management:**
- Jotai atoms are the primary state container for the React frontend (`apps/main/src/stores/jotai.ts` exports a `customStore`)
- `jotai-immer` used for mutable atom updates (e.g. `pageNumberToPageDataAtom`)
- TanStack Query handles server-state (books list, current book) with `queryKey: ["books"]` / `queryKey: ["book", id]`
- Persistent state (books metadata, auth tokens, user) lives in `tauri_plugin_store` (`store.json`)

## Key Abstractions

**Tauri Command Bridge:**
- Purpose: Type-safe IPC between React and Rust
- Examples: `apps/main/src/generated/commands.ts`, `apps/main/src/generated/types.ts`
- Pattern: Auto-generated `invoke()` wrappers; all Rust `#[tauri::command]` functions are surfaced here

**VectorStore (HNSW):**
- Purpose: Local in-process approximate nearest-neighbour search for semantic retrieval
- Examples: `apps/main/src-tauri/src/vectordb.rs`
- Pattern: Singleton `OnceLock<Arc<Mutex<VectorStore>>>` initialized lazily; backed by `.hnsw.data` / `.hnsw.graph` files in `app_data_dir`

**TTSService / TTSQueue / TTSCache:**
- Purpose: Coordinated text-to-speech pipeline with deduplication, priority queueing, and local caching
- Examples: `apps/main/src/modules/ttsService.ts`, `apps/main/src/modules/ttsQueue.ts`, `apps/main/src/modules/ttsCache.ts`
- Pattern: Singleton `EventEmitter` instances; `TTSService` is the public API, queue/cache are internal

**Extractable Trait (Rust):**
- Purpose: Common interface for book-type-agnostic data extraction
- Examples: `apps/main/src-tauri/src/shared/books.rs`
- Pattern: `trait Extractable` implemented by both `Epub` and `Pdf` structs

**Jotai Atoms:**
- Purpose: Reactive state for UI — book content, navigation, current paragraph, TTS state, user auth
- Examples: `apps/main/src/components/pdf/atoms/paragraph-atoms.ts`, `apps/main/src/components/pdf/atoms/user.ts`, `apps/web/src/atoms/state.ts`
- Pattern: Module-colocated atoms with `debugLabel` set; `customStore` used for cross-component sharing

## Entry Points

**Desktop App (Rust):**
- Location: `apps/main/src-tauri/src/main.rs` → calls `lib::run()`
- Triggers: OS launches Tauri binary
- Responsibilities: Initializes Sentry, registers all Tauri plugins, sets up SQLite connection pool (WAL mode), registers all `#[tauri::command]` handlers

**Desktop App (React):**
- Location: `apps/main/src/main.tsx`
- Triggers: Tauri WebView loads `index.html`
- Responsibilities: Creates TanStack Router with hash history, wraps app in `<Providers>`, mounts `<RouterProvider>`

**Web App:**
- Location: `apps/web/src/app/layout.tsx`
- Triggers: Next.js SSR/navigation
- Responsibilities: Wraps entire app in `<ClerkProvider>`, renders header with sign-in/sign-up/user buttons

**Web App Auth Handler:**
- Location: `apps/web/src/app/page.tsx` → `<ClerkListener>`
- Triggers: Browser navigation with `?login=true&state=...` query params
- Responsibilities: Detects Clerk sign-in, writes to Redis, redirects to Tauri deep link

**Cloudflare Worker:**
- Location: `workers/worker/src/index.ts`
- Triggers: HTTP requests from Tauri backend or web app server actions
- Responsibilities: CORS, Clerk middleware, JWT auth exchange, TTS proxy, LLM proxy, user lookups, realtime secrets

## Error Handling

**Strategy:** Rust commands return `Result<T, String>` — errors are serialized as strings and surfaced to the React layer as thrown exceptions from `invoke()`. TTS service uses `EventEmitter` error events. Worker routes return JSON error bodies with HTTP status codes.

**Patterns:**
- Rust: `.map_err(|e| e.to_string())` is the standard pattern throughout commands
- React: TanStack Query `isError` / `error` handling in route components; TTS errors emitted as `TTS_EVENTS.ERROR`
- Worker: `try/catch` blocks returning `c.json({ error: "..." }, statusCode)` on all routes
- Auth retries: `LoginButton.tsx` implements exponential backoff with up to `MAX_AUTH_RETRIES = 3` attempts

## Cross-Cutting Concerns

**Logging:** Sentry integrated in both Tauri backend (`tauri_plugin_sentry`, minidump) and Cloudflare Worker (`@sentry/cloudflare` via `Sentry.withSentry`). `console.log/error` used in React and worker for dev-time logging.

**Validation:** Zod used in Cloudflare Worker for response schema validation. State UUID validated with regex in `ClerkListener`. PKCE code_verifier/challenge used for auth proof-of-possession.

**Authentication:** JWT Bearer tokens stored in `store.json` (Tauri persistent store). All Cloudflare Worker protected routes use `requireWorkerAuth` middleware. Web app uses Clerk JWT session via `clerkMiddleware()` (Next.js middleware at `apps/web/src/middleware.ts`).
